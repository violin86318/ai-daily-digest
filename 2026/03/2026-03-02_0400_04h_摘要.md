## 🛠️ 工程与工具：MCP 已死，CLI 永生？

MCP（Model Context Protocol）曾被视为 AI 工具交互的标准协议，但工程师 EJ Holmes 撰文断言"MCP 已死"。核心论点：

- **LLM 擅长自学 CLI**：数百万 man pages、Stack Overflow、Shell 脚本的训练数据使模型能自主掌握命令行工具
- **可组合性差距**：CLI 可通过 `jq`、`grep` 管道组合，MCP 需要自定义过滤或全量加载到 context
- **调试体验**：`jira issue view` 的输出对人和模型都透明，MCP 的 JSON transport 成为黑箱
- **认证复杂性**：MCP 对认证过于"opinionated"，CLI 的 SSO/长期凭证更成熟
- **无状态优势**：CLI 是磁盘上的二进制文件，无后台进程状态管理

**结论**：最佳工具是同时服务人和机器的工具。CLI 经过数十年设计迭代，可组合、可调试、复用成熟认证系统。

**来源：** [MCP is dead. Long live the CLI](https://ejholmes.github.io/2026/02/28/mcp-is-dead-long-live-the-cli.html)

## 🛠️ 工程与工具：上下文窗口燃烧终结者

MCP 开发者 Mert Köseoğlu 发布 **Context Mode**，解决 AI 编程中的 context 消耗问题：

**核心方案**：
- **沙箱架构**：每个 `execute` 调用在独立子进程中运行，仅 stdout 进入对话 context，原始数据（日志、API 响应、快照）永不泄露
- **知识库索引**：使用 SQLite FTS5 + BM25 排名算法，对 markdown 按标题分块、代码块完整保留
- **10 种语言运行时**：JavaScript/TypeScript/Python/Shell/Ruby/Go/Rust/PHP/Perl/R，Bun 自动检测

**数据对比**：
| 场景 | 原始大小 | Context Mode |
|------|----------|-------------|
| Playwright 快照 | 56 KB | 299 B |
| 20 条 GitHub Issues | 59 KB | 1.1 KB |
| 500 行访问日志 | 45 KB | 155 B |
| 500 行 CSV | 85 KB | 222 B |
| 153 条 Git 提交 | 11.6 KB | 107 B |

**整体效果**：315 KB → 5.4 KB，**减少 98%**。Session 持续时间从 ~30 分钟延长至 ~3 小时。

**来源：** [Stop Burning Your Context Window](https://mksg.lu/blog/context-mode)

## 📚 认知与哲学：AI 时代的认知债务

Rockoder 提出**认知债务**概念，描述 AI 辅助开发中的理解危机：

**核心定义**：输出速度与理解速度之间的差距

**三大表现**：
1. **理解滞后**：prompt 秒生成数百行代码，但人类无法同速吸收
2. **评审者困境**：初级工程师生成代码速度超过高级工程师评审带宽
3. **组织记忆流失**：新工程师通过"实现的摩擦"形成隐性知识的机制被短路

**量化指标缺失**：
- 代码能工作、测试通过、功能交付 → 指标显示成功
- 理解深度无法度量 → 组织无法优化

**长期后果**：
- 3 AM 报警排查时，工程师面对"黑箱写的黑箱"
- 初级工程师永远无法形成直觉和架构判断力
- "系统正确优化了它所测量的。测量不再捕获真正重要的东西。"

**来源：** [Cognitive Debt: When Velocity Exceeds Comprehension](https://www.rockoder.com/beyondthecode/cognitive-debt-when-velocity-exceeds-comprehension/)
