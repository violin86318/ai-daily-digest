## 🔬 Designing AI for Disruptive Science
**来源**: Asimov Press | [原文链接](https://www.asimov.press/p/ai-science)

**核心观点**: 当前 AI 系统擅长在现有范式内进行预测，但无法产生颠覆性的新概念框架。

文章以 Borges 的地图作比喻：当今 AI 如同制作一幅与帝国同样详细精确的地图——完美但无用。真正需要的是像 Harry Beck 的伦敦地铁图那样的范式转换：放弃地理精确性，换取实用性。

**关键论据**:
- 19世纪电磁学由零散定律拼凑成 Maxwell 四个方程，意外预言了无线电波
- 1905年 Einstein 26岁时提出相对论，源于他作为"局外人"不受以太观念束缚
- 1980年代 Douglas Lenat 的 Automated Mathematician 看似发现新数学概念，实则只是重述了程序中隐含的内容
- Google DeepMind 的 GNoME 发现220万种新材料，但绝大多数是已知结构类型的元素替换，无法突破热力学稳定性框架

**核心洞见**: AI 擅长"超正常科学"——在当前模型内预测越来越精确，但提问全新范畴问题的能力却在萎缩。要避免这种短视，必须构建"有远见的机器"(visionary machines)，而非仅仅"预测性的机器"。

## 🤖 Cq: AI Coding Agent 的 Stack Overflow
**来源**: Mozilla AI Blog | [原文链接](https://blog.mozilla.ai/cq-stack-overflow-for-agents/)

Mozilla 正在构建 **cq**（colloquy 的缩写）——一个 AI 编码 agent 的知识共享平台。

**背景**: Stack Overflow 提问量从2014年峰值每月20万骤降至2025年12月的3,862（回到2008年发布水平）。LLM 吃掉了 Stack Overflow，而 agent 现在遇到相同问题却无人分享解决方案。

**核心设计**:
- Agent 在遇到 API 集成、CI/CD 配置等未知问题时，查询 cq commons
- 发现新知的 agent 提案，其他人确认或标记过时
- 信任通过使用而非权威建立
- 84%开发者使用 AI 工具，但46%不信任准确性（同比上升）

**技术栈**: Claude Code 插件、OpenCode 插件、MCP server、团队 API、人工审查 UI

**开源**: [github.com/mozilla-ai/cq](https://github.com/mozilla-ai/cq)

## 📞 AI 接待员实战：豪华修车厂案例
**来源**: It That Lady Dev | [原文链接](https://www.itsthatlady.dev/blog/building-an-ai-receptionist-for-my-brother/)

开发者为弟弟的豪华修车厂构建 AI 接待员 **Axle**，解决每周100+未接来电、每月数千美元收入损失的问题。

**技术架构**:
- **Vapi** — 电话号码、语音转文字、文字转语音（ElevenLabs）、实时 function calling
- **FastAPI + Ngrok** — Webhook 服务器
- **MongoDB Atlas** — 知识库存储 + 向量搜索
- **Voyage AI (voyage-3-large)** — 1024维文本嵌入
- **Claude Sonnet-4** — 知识库增强的回答生成

**核心经验**:
1. 绝不用原始 LLM 做商业语音 agent，必须用 RAG grounding
2. 语音回复需要完全重写系统 prompt：短句、无 markdown、无"好的问题！"等填充词
3. 升级流程（未知问题时收集回调）是核心功能，不是边缘 case
